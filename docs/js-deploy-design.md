## 1. Repo summary

### 1.1 `zyndai-ts-sdk` CLI output shape

The TS SDK ships `zynd` (Commander, fully bundled CJS at `dist/cli/index.js`). `zynd agent init` and `zynd service init` scaffold a flat directory containing:

- `agent.config.json` or `service.config.json` with `name`, `framework`, `language: "ts"`, `description`, `category`, `tags`, `summary`, `registry_url`, `webhook_port`, `entity_index`. Written by `src/cli/agent.ts:155-169` and `src/cli/service.ts:121-133`.
- `agent.ts` or `service.ts` (entry, copied from `src/templates/ts/<framework>.ts.tpl`).
- `payload.ts` (Zod request/response schema, from `src/templates/ts/payload.ts.tpl`).
- `package.json` with `"type": "module"`, scripts `start`, `dev`, `typecheck`, `dependencies` including `zyndai`, `dotenv`, `zod` plus framework deps, `devDependencies` `tsx` `^4.19.0`, `typescript` `^5.7.0`, `@types/node` `^22.0.0`. Written by `src/cli/scaffold-ts.ts:53-99`.
- `tsconfig.json` (ESM, `target: ES2022`, `module: ESNext`, `moduleResolution: Bundler`, `noEmit: true`).
- `.gitignore` (`node_modules/`, `dist/`, `.env`, `.env.local`).
- `.well-known/agent.json` placeholder (regenerated by SDK on start).
- `.env` with `ZYND_AGENT_KEYPAIR_PATH` (or `ZYND_SERVICE_KEYPAIR_PATH`) and `ZYND_REGISTRY_URL` plus blank framework-API-key lines (`OPENAI_API_KEY`, `TAVILY_API_KEY`, etc.).

The keypair lives **outside** the project at `~/.zynd/agents/<slug>/keypair.json` (or `services/<slug>/...`), referenced by absolute path in `.env`. There is no compiled output — `package.json#scripts.start` is `zynd agent run`, which spawns `npx tsx agent.ts` (`src/cli/agent.ts:319-337`). Node 18+ required (`package.json:60-62`). Frameworks supported: `langchain`, `langgraph`, `crewai`, `pydantic-ai`, `vercel-ai`, `mastra`, `custom` (`src/templates/frameworks.ts:22-65`).

The SDK package itself uses **npm** in install commands shown to users (e.g. `src/templates/frameworks.ts:26`); there is no pnpm/yarn lockfile committed and `package.json` doesn't pin a package manager via `packageManager`. So the package manager for **scaffolded projects** is `npm` by convention. The SDK's own dev environment uses pnpm (per `pnpm-lock.yaml` in repo root, see commit `ead261f`), but that's irrelevant to user output.

### 1.2 `zynd-deployer` current architecture

Next.js 15 monolith + sibling Node worker process, both in TS, single VM. Source paths cited:

- **UI/API**: `src/app/` — Next App Router. Pages: `/` (list), `/deploy` (drag-drop wizard, `src/app/deploy/page.tsx`), `/d/[id]` (detail). API routes: `POST /api/deployments` (multipart upload, `src/app/api/deployments/route.ts:46-169`), `GET /api/deployments`, `GET/DELETE /api/deployments/[id]`, log endpoints.
- **Worker**: `worker/main.ts` polls Postgres every 1s. State machine in `worker/lifecycle.ts:64-183` walks `queued -> unpacking -> writing_config -> allocating_port -> starting -> health_checking -> registering_route -> running`.
- **Persistence**: Postgres via Prisma (`prisma/schema.prisma`). `Deployment`, `DeploymentLog`, `DeploymentMetric`, `PortAllocation` tables. No `User`.
- **Deploy mechanism**: not k8s, not ssh, not a remote registry. `dockerode` calls a local `/var/run/docker.sock` and bind-mounts `/var/lib/zynd-deployer/work/<id>` into the container at `/app:ro` (`worker/docker.ts:64-91`). The base image is built once on the VM by `infra/install.sh`; per-deploy we just `docker run` against it. User code never gets baked into a custom image.
- **Runtime detection**: hardcoded. `validator.ts:168-179` accepts only `agent.config.json` + `agent.py` or `service.config.json` + `service.py`. `docker.ts:67-70` runs `python /app/agent.py` or `python /app/service.py`.
- **Routing**: Caddy admin API (`worker/caddy.ts`), wildcard `*.deployer.zynd.ai` over DNS-01.
- **Secrets**: agent keypair age-encrypted at rest (`src/lib/crypto.ts`), decrypted to a tmpfs path only at `docker run`. Developer key never uploaded.
- **Env injection**: rewritten merge in `worker/lifecycle.ts:194-248`. Deployer sets `ZYND_AGENT_KEYPAIR_PATH=/app/keypair.json`, `ZYND_REGISTRY_URL`, `ZYND_WEBHOOK_PORT=5000`, `ZYND_ENTITY_URL=https://<slug>.deployer.zynd.ai`. Original `.env` keys preserved.

The Python path is mature, has crash watching, log streaming via SSE, periodic health probes, and metric sampling. JS deploy must inherit all of this for free where possible.

## 3. Code structure changes in `zynd-deployer`

### 3.1 The abstraction shape

Three options on the table:

**A. Strategy pattern (`Runtime` interface) inside `worker/`.** One `Runtime` interface with `validate`, `installCommand`, `startCommand`, `baseImage`, plus glue methods. Two implementations: `PythonRuntime`, `NodeRuntime`. Worker code is generic; `lifecycle.ts` looks up the runtime once per deployment by reading `Deployment.runtime`.

**B. Two parallel modules (`worker/python/`, `worker/node/`).** Each is a near-copy of the current code. Routed by a switch in `lifecycle.ts`.

**C. Adapter / plug-in registry.** Out-of-process plug-ins (overkill - rejected immediately).

**Pick A.** The diff between Python and JS deploys is small (4 forks, listed above). The lifecycle, port allocation, Caddy registration, log tailing, crash detection, and health probing are identical. Forking the whole worker for one language is duplicate maintenance and a guaranteed source of drift. Strategy keeps the state machine generic; `Runtime` is the only language-specific surface.

The interface is small enough that "build for two before generalizing" doesn't bite us - we *have* two now (Python today, JS next), and the abstraction barely exists beyond a config table plus one `runContainer` arg. It's not a premature framework.

## 4. Dockerfile / runtime template for JS

Decisions baked in:

- **Node version: `node:22-slim`** (current LTS as of 2026-04). Matches `engines.node >= 18` from the SDK and the deployer's own `>=20`.
- **Package manager: npm.** Scaffolded TS projects are produced by `src/cli/scaffold-ts.ts:96` writing only `package.json` (no `pnpm-lock.yaml`, no `yarn.lock`). The SDK README and template install commands all use `npm install` (`src/templates/frameworks.ts:26-64`). Standardising on npm in the base image avoids a per-deploy package-manager probe and keeps cold-start cheap. We can layer in pnpm-via-corepack later if users start shipping pnpm-lock'd projects.
- **No build step.** Run `agent.ts` directly with `tsx`, same as `zynd agent run` does locally. This keeps deploy behaviour identical to dev behaviour, sidesteps `tsc` failures from strict-mode user code, and removes the temptation to invent a `dist/` artifact contract. The cost is `tsx` JIT-compilation on every container start, which adds <500ms - well under the existing 15s cold-start budget for Python.
- **Start command: `tsx /app/<entry>.ts`** (or fall back to `node /app/<entry>.js` if a precompiled JS entry exists). The Runtime adapter resolves entry order: `agent.ts -> agent.js -> agent.mjs -> agent.cjs`. Mirrors what `src/cli/agent.ts:311-313` does locally.
- **Framework SDKs pre-installed in the base image.** Same trade-off Python made: one ~1.5GB fat image keeps cold-start under 15s for the common case (`PLAN.md` open question 5). Pre-install the same set the TS framework registry advertises: `langchain`, `@langchain/openai`, `@langchain/community`, `@langchain/core`, `@langchain/langgraph`, `ai`, `@ai-sdk/openai`, `@mastra/core`, `zod`, `dotenv`, `tsx`. User's own `npm install` happens inside `entrypoint-node.sh` only if `package-lock.json` is present and only adds packages missing from the base layer.
- **Healthcheck: none baked into the image.** The deployer pings `/health` from outside via `worker/lifecycle.ts:waitForHealth`, identical to Python. Don't add a Docker `HEALTHCHECK` line - it just creates a parallel signal that drifts from the source-of-truth probe.

Skeleton of `images/Dockerfile.node-agent-base` (sketch):

```
FROM node:22-slim
ENV NODE_ENV=production
RUN apt-get update && apt-get install -y --no-install-recommends curl ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# Pre-install the fat framework set into a global, side-loadable path so
# user projects that don't ship their own copy can resolve them.
RUN mkdir -p /opt/zynd-base && cd /opt/zynd-base \
    && npm init -y >/dev/null \
    && npm install --omit=dev \
       zyndai dotenv zod tsx \
       @langchain/openai @langchain/community @langchain/core langchain \
       @langchain/langgraph \
       ai @ai-sdk/openai \
       @mastra/core
ENV NODE_PATH=/opt/zynd-base/node_modules

WORKDIR /app
EXPOSE 5000
COPY images/entrypoint-node.sh /usr/local/bin/zynd-entrypoint
RUN chmod +x /usr/local/bin/zynd-entrypoint
ENTRYPOINT ["/usr/local/bin/zynd-entrypoint"]
CMD ["node", "/opt/zynd-base/node_modules/.bin/tsx", "/app/agent.ts"]
```

`entrypoint-node.sh` (sketch):

```
#!/usr/bin/env bash
set -euo pipefail
EXTRA_DIR="/tmp/user-modules"
if [[ -f /app/package-lock.json ]]; then
  mkdir -p "$EXTRA_DIR"
  cp /app/package.json /app/package-lock.json "$EXTRA_DIR"/
  ( cd "$EXTRA_DIR" && npm ci --omit=dev --no-audit --no-fund )
  export NODE_PATH="$EXTRA_DIR/node_modules:$NODE_PATH"
elif [[ -f /app/package.json ]]; then
  mkdir -p "$EXTRA_DIR"
  cp /app/package.json "$EXTRA_DIR"/
  ( cd "$EXTRA_DIR" && npm install --omit=dev --no-audit --no-fund )
  export NODE_PATH="$EXTRA_DIR/node_modules:$NODE_PATH"
fi
exec "$@"
```

Notes:

- We install into `/tmp/user-modules` because `/app` is bind-mounted read-only (mirroring `worker/docker.ts:78` `:ro` for Python). Tmpfs at `/tmp` is already allocated by the existing Python container config (`worker/docker.ts:85`).
- `NODE_PATH` chaining means `import "zyndai"` resolves to the user's pinned version if they shipped one, falling back to the base layer otherwise. This matches Python's `PYTHONPATH` chain in `images/entrypoint.sh:22`.
- `--omit=dev` skips `tsx`/`typescript`/`@types/node` from the user's package.json since they're already in the base. Cuts ~50MB and ~5s off cold-start.

`Dockerfile.node-service-base` is the same minus `langchain`/`@langchain/*`/`@mastra/core`/`ai` - services rarely need them, mirroring the Python service-base trim.

## 5. UI design

### 5.1 The two options

**(a) Language-first wizard (`/deploy` shows JS vs Python first, then Agent vs Service).** Two clicks to start a deploy.

**(b) Language toggle on existing tab screen (current `/deploy` keeps Agent/Service tabs, adds a JS/Python toggle next to them).** One screen, two orthogonal toggles.

**Recommendation: (b), the toggle.** Reasoning:

- The current screen is already a two-mode UI (Agent/Service tabs, `src/app/deploy/page.tsx:20-34`). A 2x2 selection (lang x type) is genuinely a 2D choice, and forcing it into a sequential wizard adds a click without information gain - the user already knows what they want when they hit `/deploy`.
- Wizard pattern earns its keep when later steps depend on earlier ones. Here they don't: the validation rules and copy change slightly, but the actual UI affordances (project drop zone, keypair drop zone, deploy button) are identical.
- The language toggle exposes the choice without nesting it. Users discovering the deployer for the first time see "we support both" at a glance.
- Most important: switching language doesn't require a screen change. State is local to one component.

The wizard still wins if we expect to add a 3rd or 4th language and the per-language UI starts diverging materially (e.g. JS shows a "framework picker" Python doesn't have). Re-evaluate at that point.

## 7. Risks and open questions

1. **Language detection.** Two project layouts could ambiguously match: a Python project that happens to have a `package.json` (e.g. for a CDN-bundled UI), or a TS project whose user wrote a sidecar `agent.py` for tests. Recommendation: **use the explicit `language` field that already exists in `agent.config.json` / `service.config.json`** (the TS scaffolder writes `language: "ts"` at `src/cli/agent.ts:160`; the Python scaffolder doesn't write a `language` key today). Fallback: presence of `agent.py`/`service.py` -> python; presence of `package.json` with `dependencies.zyndai` and an `agent.ts`/`service.ts` -> node. Reject ambiguous uploads with a clear error rather than guessing. Have the UI also pass the user's explicit toggle as a hint - server still validates that the upload matches the hint.

2. **Native deps (`node-gyp`).** A user shipping `better-sqlite3`, `bcrypt`, or any package with a binary build step will fail `npm install` at container start because the slim base image lacks `python3`/`make`/`g++`. Options: (a) add the build toolchain to the base image (~150MB extra), (b) document the limitation and tell users to use pure-JS alternatives, (c) detect from `package-lock.json` whether any dep has `bin` scripts and fall back to a "fat" image. Recommendation for v1: **(b) document and reject at validate-time** if user's `package.json` declares a known-native-dep against an allowlist (small list, easy to maintain).

3. **Cold-start cost of `npm install`.** Even with the base layer covering frameworks, a user with 30 transitive deps not in the base image hits `npm install` cold (no cache, slim image, network IO). Could blow past the 30 retries x 500ms health budget. Mitigation: bump health-check window to 60s for `runtime=node`, OR persist a `/var/lib/zynd-deployer/npm-cache` volume shared across deploys via `--mount type=bind,source=...,target=/root/.npm`. Recommendation: ship with **the longer health window first** (cheaper change), add the shared cache in v2 if data shows it's needed.

4. **Monorepo / workspace deps.** A user dragging a folder out of a pnpm/npm workspace may have a `package.json` with `workspace:*` deps that `npm install` can't resolve. We can't detect every flavor. Validate-time check: scan `package.json` for `"workspace:"` substrings in any dep value, reject with a helpful message ("flatten your project before deploying"). 

5. **`tsx` vs `node` choice.** Running TS via `tsx` adds ~300ms startup and re-transpiles on every cold start. Some users may ship a pre-built `dist/index.js`. Recommendation: prefer compiled `agent.js`/`service.js` if present; fall back to `agent.ts`. The runtime adapter does this lookup once at container start. Documented in 4 above.

6. **HD derivation prefix mismatch (active cross-SDK bug, per memory).** TS uses `zns:agent:`, Python uses `agdns:agent:`. The deployer doesn't currently care - it never derives a keypair, only encrypts and bind-mounts the user-supplied one - so the deployer is **not affected**. But if anyone in the deploy validator ever computes `entity_id` from the public key to cross-check the registry, it must use the same prefix the SDK that **generated** the key used. Flag for the validator implementer: don't derive, just trust the uploaded keypair as the SDK already did.

7. **Slug collisions across runtimes.** If a user deploys `weather-bot` in Python and later `weather-bot` in JS, the slug allocator gives them the same slug (uniqueness today is on `name` only via `slugify`, see `src/lib/slug.ts`). With our changes the existing collision logic in `route.ts:108-115` still triggers - **no new risk**, but worth confirming the operator-facing error message reads sensibly when the conflict is across runtimes. Probably reuse as-is.

8. **Manifest size cap.** Current Python rule: 200 lines in `requirements.txt`. JS analog: total dep count in `package.json` `dependencies` + `devDependencies`. Pick a number proportional to install time, not line count. Suggest cap at 100 direct deps. (`devDependencies` are ignored at runtime via `--omit=dev` so are mostly cosmetic.)

9. **Open: how does the deployer surface SDK version drift?** A user's `package.json` may pin `zyndai@0.1.0` while the base image preinstalls `0.2.x`. With our `NODE_PATH` chain, the user's pinned copy wins - good. But the deployer doesn't currently log the resolved version. Add a one-line system log entry post-install: `[worker] zyndai resolved to <version>` so debugging is easy. Cheap, useful.

## 8. Phased rollout

### Slice 1 - "JS deploy works for one framework, mirroring Python flow" (MVP)

1. Schema: add `runtime` column + migration, default `"python"`.
2. `src/lib/runtime.ts` + `src/lib/detect.ts` + extract `worker/runtimes/python.ts` (no behaviour change - pure refactor first, ship it standalone, verify no Python regressions).
3. `worker/runtimes/node.ts` covering only `custom` framework (the simplest TS template, smallest dep surface).
4. `images/Dockerfile.node-agent-base` and `node-service-base` with **only** `zyndai`, `dotenv`, `zod`, `tsx` pre-installed. No LangChain/Mastra in the base yet.
5. Validator: accept JS uploads (`package.json` + `agent.ts`/`service.ts`).
6. UI: language toggle on `/deploy` page only (no detail-page badge, no list-page column - cosmetic).
7. End-to-end smoke: deploy `examples/simple-service.ts` (already in `examples/` per `REPO_MAP.md:107`), confirm it registers on `dns01.zynd.ai`.

This slice ships the architectural change (Runtime strategy) plus the thinnest-possible JS path. Operationally it proves the abstraction is real before we spend the time pre-installing the framework matrix.

### Slice 2 - Framework support and ergonomics

1. Bake the full TS framework set (LangChain.js, LangGraph.js, Vercel AI SDK, Mastra, etc.) into the agent base image.
2. Accept and respect `package-lock.json` (`npm ci` over `npm install`).
3. Native-dep allowlist + reject in validator.
4. Detail page badge + list page column for runtime.
5. Bump JS health-check window to 60s.

### Slice 3 - Sharper edges (deferred)

- Shared `/root/.npm` cache volume across deploys (only if cold-start data justifies it).
- Pre-built JS support (`dist/index.js` precedence over `agent.ts`).
- Runtime-specific resource caps (Node containers may need more memory than Python's 1536MB default for users running heavy LangChain.js workloads - tune via `runtime.resourceHints`).
- pnpm support via Corepack if user demand emerges.

## 9. Recommendation summary

- Strategy pattern (single `Runtime` interface) over forking the worker - the overlap is too big to duplicate.
- Two new base images, npm only, no build step (run `tsx agent.ts` directly, mirroring `zynd agent run`).
- Language toggle on the existing `/deploy` page over a wizard - the choice is 2D, not sequential.
- Env contract is unchanged - the TS templates already read the exact same `ZYND_*` names the deployer injects.
- Ship the refactor + thinnest possible JS path as Slice 1; expand framework support and ergonomics in Slice 2.

## 10. Bets this design makes

- npm is "good enough" as the only supported package manager. Risk: low; users can pre-install via Docker layer if they want pnpm.
- `tsx` cold-start is acceptable inside the existing health budget. Risk: medium for projects with heavy import graphs; mitigated by Slice 2 health-window bump.
- The base image fat-set keeps cold-starts under 15s for the common case. Risk: medium - same bet Python's already running on. If we hit it, slice the image per-framework as v2.
- Users will not commonly need native build deps. Risk: medium; mitigated by validator allowlist + clear error message.
- The Runtime abstraction stays small. Two implementations is the threshold where generalising is honest, not premature; stay disciplined about not adding `extension hooks` until a third runtime asks for them.
